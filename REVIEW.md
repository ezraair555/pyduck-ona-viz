# Code Review: `pyduck-ona-viz`

**Repo:** https://github.com/ezraair555/pyduck-ona-viz (local clone reviewed)
**Version reviewed:** 0.1.0
**Date:** 2026-06-27

---

## Final Grade

**C+ — 76 / 100**

**Verdict: FAIL** (below the required A bar for a publication-ready public release)

The package is usable and the visual output is attractive, but it is not yet production-ready. The most serious issues are: a false claim of "103 tests" when only 14 tests exist, no CI/GitHub Actions, no `.gitignore`, no input sanitization before embedding data into HTML/JS strings, and widespread type-safety gaps that make the `py.typed` marker misleading. These are all fixable in a focused pass.

## 1. Documentation (25 pts) — 18 / 25

### Strengths
- `README.md` is well-structured, explains the companion relationship to `pyduck-ona`, lists every public function, and documents the design language and API patterns.
- Every public function has a docstring, and most use **NumPy-style** sections (`Parameters`, `Returns`).
- `__init__.py` includes a clear module-level docstring enumerating the 10 entry points.
- Examples in the README are concrete and copy-pasteable.

### Issues
- **False test claim in README context.** The context notes "103 tests passing locally," and the README says `pytest tests/` verifies every function. In reality there are **14 tests**. This is a credibility problem for a public release, even if the README itself does not literally print "103."
- **Docstring conventions are inconsistent.** Public functions use NumPy-style, but internal helpers often have no docstring at all (e.g., `_span_of_control_plotly` in `span_control.py:175`, `_silo_map_matplotlib`/`_silo_map_pyvis` in `network_viz.py:230`, `_wrap_html` in `dashboard.py:225`).
- **Return types in docstrings are vague.** Many say `matplotlib.figure.Figure` but the actual annotation is `Any` (`span_of_control.py:42`, `hr_viz.py:29`, `network_viz.py:45`).
- **Missing API reference.** There is no `docs/` folder, no Sphinx/MkDocs site, no readthedocs/GitHub Pages link. For a "publication-quality" package, autogenerated API docs are expected.
- **No changelog or release notes.** First release should at least have a `CHANGELOG.md` or GitHub release notes.

### Citations
- `README.md:121-122` — "Smoke tests verify that every function returns the correct type." Only 14 smoke tests exist; no pixel, property, or edge-case tests.
- `src/pyduck_ona_viz/__init__.py:31-41` — good module docstring.
- `src/pyduck_ona_viz/span_control.py:175` — `_span_of_control_plotly` has no docstring.
- `src/pyduck_ona_viz/network_viz.py:230,282` — `_silo_map_matplotlib` and `_silo_map_pyvis` have no docstrings.

## 2. Testing (25 pts) — 10 / 25

### Strengths
- The existing 14 tests are fast (≈2 s), run cleanly, and exercise every public entry point at least once.
- The synthetic fixture in `tests/test_viz.py` is reusable and reasonably realistic.
- One test checks that bad input raises (`test_invalid_input_raises`).

### Issues
- **Massive test-count discrepancy.** Context claims 103 tests; actual count is **14**. Running `pytest tests/ -v` collected 14 items and all 14 passed. This is not a rounding error — it is an 86% gap.
- **No coverage tooling configured.** `pyproject.toml` has no `pytest-cov` or coverage threshold.
- **Very few edge cases.** Missing tests for:
  - Empty DataFrames / zero-row input.
  - Single-row input.
  - Duplicate employee IDs in hierarchy.
  - Cycles in the reporting chain (`reporting_chain_walk`).
  - `root_id` not found in `org_chart_tree`.
  - `color_by="level"` and `color_by=None` for `org_chart_tree`.
  - `metadata` missing `id_col`.
  - `return_html=True` for `silo_map` and `span_of_control` only checks string contains "plotly".
  - HTML output validity / no `JS injection` via employee names.
  - `summary_dashboard` without optional frames.
  - `attrition_heatmap` raw vs pre-aggregated branches.
- **No parametrization.** `pytest.mark.parametrize` is never used; every function has one happy-path test.
- **No fixture scope rationale.** The module-scoped `synthetic` fixture mutates nothing, which is fine, but it is large and shared across unrelated tests.
- **Test docstring overpromises.** `tests/test_viz.py:6-10` says tests catch "broken column-name contracts" — but only one negative test exists and it does not cover all functions.

## 3. Code Review (30 pts) — 22 / 30

### Strengths
- The public API is clean and the package is well-organized into modules by visualization type.
- Color palette, typography, and layout defaults are centralized in `theme.py`.
- Input column validation is generally present (`KeyError` / `TypeError`).
- The D3 org chart is a genuinely useful standalone HTML artifact.

### Issues

#### Type hints & mypy
- `py.typed` is present (`src/pyduck_ona_viz/py.typed`), but `mypy src` reports **37 errors** under the project's own strict configuration. This makes the typed-package claim misleading.
- Major recurring errors:
  - `Function is missing a return type annotation` on internal helpers (`dashboard.py:127`, `dashboard.py:147`, `dashboard.py:166`, `dashboard.py:185`, `dashboard.py:225`, `hr_viz.py:174`, `network_viz.py:230`, `network_viz.py:282`, `theme.py:208`).
  - `Function is missing a type annotation for one or more parameters` (`theme.py:137`, `theme.py:148`, `theme.py:208`, `theme.py:229`, `theme.py:254`).
  - `List comprehension has incompatible type List[tuple[float, float, float, float]]; expected List[str]` in `theme.py:181` and `theme.py:201` — `to_rgba` returns RGBA tuples, not hex strings, but the return annotation says `list[str]`.
  - `Incompatible types in assignment` in `span_control.py:130` (`bar_colors` alternates `str` and `list[str]`), `network_viz.py:112`, and `hr_viz.py:248`.
  - `Dict entry ... has incompatible type` in `org_chart.py:67` and `org_chart.py:114` due to overloaded dict usage.
  - Missing stubs for pandas, networkx, plotly, pyvis. These should be declared in `[tool.mypy]` ignore or stub dependencies added.

#### Security — untrusted HTML/JS injection
- **Critical:** User-supplied strings are interpolated directly into HTML and JavaScript without escaping.
  - `org_chart.py:158-162` — `title`, `width`, `height`, `color_by` inserted into HTML/CSS/JS with only `title` escaped; `width`/`height` and `color_by` are not escaped.
  - `org_chart.py:263-265` — `tree_json` and `legend_json` are JSON-encoded, which is safe for JS string contexts.
  - `org_chart.py:247` — node `name`, `title`, `department`, `level` are not escaped before being inserted into JS strings via JSON. JSON encoding handles quotes but not all XSS vectors; `</script>` inside a name could break the script block.
  - `network_viz.py:319-322` — `_silo_map_pyvis` wraps `net.generate_html()` and inserts `title` and `subtitle` directly into HTML; neither is escaped.
  - `dashboard.py` — `_wrap_html`, `_panel_card`, `_kpi_cards_html` insert `title`, `subtitle`, and arbitrary Plotly div HTML without escaping. If a user passes a KPI string containing `<script>`, it executes.
  - `span_control.py:197-199` — Plotly title HTML includes `subtitle` without escaping.
- Recommendation: pass all user strings through `html.escape()` before interpolating, and avoid building HTML via f-strings for dynamic content.

#### Error handling
- `org_chart_tree` does not validate `root_id`. If `root_id` is not in `children_map`, `make_node(chosen_root)` returns a node with no children and no error (`org_chart.py:256-258`).
- `reporting_chain_walk` silently truncates cycles (`org_chart.py:308-312`) instead of warning or raising. A cycle in hierarchy data is a data-quality bug that should be reported.
- `silo_map` silently ignores `partition_col` values that are NaN, which is acceptable but undocumented.
- `_normalize_hierarchy` does not check for duplicate rows/duplicate employee IDs, which can produce malformed org charts.

#### Performance footguns
- Several functions iterate over DataFrames row-by-row via `iterrows()` to build lookups:
  - `span_control.py:80-86`
  - `network_viz.py:100-108`
  - `org_chart.py:244-246`
  - `org_chart.py:300-304`
- For modest org sizes this is fine, but for 100k+ rows this is a real footgun. Vectorized merge/apply or `set_index().to_dict()` should be preferred.
- `hierarchy_depth_heatmap` (`hierarchy_viz.py:88-92`) loads the whole wide matrix into memory via `.to_numpy()`; acceptable given the use case but worth noting for very deep orgs.

#### Dead / misleading code
- `tests/test_viz.py:121-127` — the pivot-based wide-frame construction is immediately overwritten by a literal `wide` DataFrame. The pivot block is dead code.
- `network_viz.py:34-41` — `_CENTRALITY_PANELS` is defined but never used; `centrality_dashboard` builds its own frame list at line 55.
- `span_control.py:47` — `_prepare_stats` accepts `label_col` parameter but does not use it; the actual label resolution happens in `span_of_control` at line 103.
- `theme.py:181-188` — `gradient_color` returns RGBA tuples under a `list[str]` annotation and is not used anywhere in the package.

#### API consistency
- Public function defaults vary subtly:
  - `span_of_control` defaults `id_col="manager_id"`, but most other functions default to `"employee_id"`.
  - `compensation_equity` defaults `group_col="gender"`, which is opinionated; acceptable but should be documented.
- `centrality_dashboard` takes four separate DataFrames instead of a single combined frame; this matches pyduck-ona output but makes the API rigid.

#### Imports
- `__init__.py` imports correctly and `__all__` is complete. No incorrect imports detected.
- Several modules use local/late imports inside functions (`plotly.graph_objects`, `networkx`, `pyvis.network`, `matplotlib.cm`, `matplotlib.patches`). This is defensible to keep optional dependencies optional, but it scatters dependency expectations and hurts static analysis.

### Citations
- `src/pyduck_ona_viz/py.typed` — typed marker present but `mypy` fails.
- `src/pyduck_ona_viz/theme.py:181,201` — `gradient_color` / `gradient_color_palette` return type bug.
- `src/pyduck_ona_viz/span_control.py:130` — `bar_colors: list[str] | str` inconsistency.
- `src/pyduck_ona_viz/org_chart.py:158-162,247,319-322` — XSS risk from unescaped user strings.
- `src/pyduck_ona_viz/org_chart.py:308-312` — cycle handling silent truncation.
- `src/pyduck_ona_viz/network_viz.py:34-41` — dead `_CENTRALITY_PANELS`.
- `src/pyduck_ona_viz/dashboard.py:127,147,166,185,225` — missing return annotations.

## 4. Packaging & Repo Hygiene (10 pts) — 4 / 10

### Strengths
- `pyproject.toml` is present and largely complete: build backend, project metadata, optional dependencies, project URLs, classifiers, and tool configs for black/ruff/mypy/pytest.
- `LICENSE` is present and correctly placed at repo root.
- `py.typed` marker is packaged via `[tool.setuptools.package-data]`.

### Issues
- **No `.gitignore`.** Running the demo produced `tests/__pycache__/` as an untracked directory. Without a `.gitignore`, build artifacts, caches, and `examples/output/` will be committed accidentally. Already `examples/output/01_span_of_control.html` and `examples/output/04_centrality_dashboard.png` show modifications in `git status`.
- **No CI / GitHub Actions.** There is no `.github/workflows/` directory. A public package claiming 103 tests and typed code needs automated testing on PRs, plus lint/type jobs.
- **No pre-commit config.** `pyproject.toml` configures ruff/black/isort/mypy, but nothing enforces them.
- **No `MANIFEST.in`.** Relying solely on `tool.setuptools.package-data` is okay, but a `MANIFEST.in` would make source distribution contents explicit.
- **Version pinning is loose.** Dependencies use `>=` floors with no upper bounds. Fine for a young package, but matplotlib/plotly major releases can break visual output; consider capping at next major or testing against pre-releases in CI.
- **`.egg-info/` is tracked/committed.** `src/pyduck_ona_viz.egg-info/` appears in the file tree; it should be ignored.
- **`[full]` extra installs `pyduck-ona>=0.1`,** which may or may not exist on PyPI; no lock file or test for the `[full]` install path.

## Top 7 Fixes to Reach A Grade

1. **Correct the testing story (P0).**
   - Add tests until the suite actually covers the 103-test claim, or remove the misleading claim from the README/context.
   - Add parametrized edge-case tests: empty frames, cycles, bad `root_id`, XSS payloads, single-row inputs, missing optional frames.
   - Add `pytest-cov` and enforce a threshold (e.g., 90%).

2. **Add CI / GitHub Actions (P0).**
   - Create `.github/workflows/ci.yml` running pytest, ruff, black, isort, and mypy on Python 3.10–3.12.
   - Add a `.gitignore` and remove `src/pyduck_ona_viz.egg-info/` from tracking.

3. **Fix HTML/JS injection vulnerabilities (P0).**
   - Escape every user string inserted into HTML/JS in `org_chart.py`, `network_viz.py`, and `dashboard.py` using `html.escape()`.
   - Add a test that passes a name like `<script>alert(1)</script>` and verifies it is rendered as text.

4. **Fix mypy / type annotations (P1).**
   - Add missing return and parameter annotations to internal helpers.
   - Fix `gradient_color` / `gradient_color_palette` return types or remove the unused `gradient_color`.
   - Add stub dependencies (`pandas-stubs`, `types-networkx`) and configure mypy to ignore missing stubs for optional packages (plotly/pyvis) or vendor minimal stubs.

5. **Add a `.gitignore` and commit hygiene (P1).**
   - Ignore `__pycache__/`, `.pytest_cache/`, `*.egg-info/`, `dist/`, `build/`, `.venv/`, and `examples/output/` (or regenerate them in CI and attach as artifacts).

6. **Improve error handling and validation (P2).**
   - Validate `root_id` exists in `org_chart_tree`.
   - Warn/raise on cycles in `reporting_chain_walk`.
   - Validate unique employee IDs in hierarchy inputs.

7. **Build real API documentation (P2).**
   - Add a Sphinx or MkDocs site with `autodoc`/`mkdocstrings` and link it from the README.
   - Replace "screenshot placeholder" text with embedded committed PNGs.
   - Add a `CHANGELOG.md`.

## Summary

`pyduck-ona-viz` produces genuinely nice-looking visualizations and has a coherent public API, but it cannot be called publication-ready in its current state. The gap between claimed and actual test coverage, the absence of CI, the lack of a `.gitignore`, and the HTML-injection vulnerabilities are release-blocking. Fix items 1–3 and the package would jump to a solid B; fix all seven to reach A.

**Final grade: C+ (76 / 100) — Verdict: FAIL.**
